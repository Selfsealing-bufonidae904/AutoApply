## 1. Purpose and Scope

### 1.1 Purpose
Specifies all functional and non-functional requirements for the Resume Versioning
feature (v2.1.0). Audience: System Engineer, Backend/Frontend Developers, Testers.

### 1.2 Scope
The system SHALL provide a resume library screen, resume detail view with metadata,
effectiveness metrics, PDF preview, and download capability for all AI-generated
resume versions.

The system SHALL NOT provide side-by-side comparison, in-app editing, tagging, or
cover letter versioning in this release.

### 1.3 Definitions and Acronyms

| Term              | Definition                                                    |
|-------------------|---------------------------------------------------------------|
| Resume Version    | A tailored resume PDF+MD generated by the AI engine for a specific job application |
| Fallback Resume   | The user's static resume used when no AI provider is configured |
| Effectiveness     | The correlation between resume usage and application outcome (interview/offer) |
| Resume Library    | A dedicated UI screen listing all generated resume versions   |
| Match Score       | Numeric 0-100 score from job filtering indicating fit quality |

---

## 2. Overall Description

### 2.1 Product Perspective
Resume Versioning connects existing but disconnected data: resume files on disk
(`~/.autoapply/profile/resumes/`) and application records in SQLite. It surfaces
this data through a new UI screen and API endpoints.

### 2.2 User Classes

| User Class        | Description              | Frequency   | Technical Expertise |
|-------------------|--------------------------|-------------|---------------------|
| Active Applicant  | Daily job seeker          | Daily       | Novice              |
| Strategy Optimizer| Weekly outcome reviewer   | Weekly      | Intermediate        |

### 2.3 Operating Environment
- Electron desktop app (Windows, macOS, Linux)
- Flask backend with SQLite database
- Vanilla JS ES modules frontend
- Resume files stored as PDF+MD on local filesystem

### 2.4 Assumptions

| # | Assumption | Risk if Wrong | Mitigation |
|---|-----------|---------------|------------|
| A1 | Resume files remain at paths stored in `applications.resume_path` | Files not found | Show graceful "file missing" state |
| A2 | All generated resumes have both .md and .pdf variants | Missing markdown | Fall back to PDF-only display |
| A3 | Application status accurately reflects outcome | Metrics incorrect | Documented as data quality dependency |
| A4 | A new `resume_versions` table is acceptable | Schema change concern | Additive-only, no ALTER on existing tables |

### 2.5 Constraints

| Type       | Constraint                                  | Rationale                    |
|------------|---------------------------------------------|------------------------------|
| Technical  | No new Python dependencies                  | Bundle size control          |
| Technical  | No modification to existing `applications` table | Backward compatibility    |
| Technical  | Vanilla JS ES modules, no build step        | ADR-017                      |
| Technical  | All strings via `t()` / `data-i18n`         | i18n requirement             |
| Technical  | WCAG 2.1 AA for all new UI elements         | Accessibility requirement    |

## 3. Functional Requirements

### FR-110: Resume Version Storage

**Description**: The system shall store metadata about each generated resume version
in a dedicated database table, capturing the target job, generation context, and file paths.

**Priority**: P0
**Source**: US-090, US-091
**Dependencies**: None

**Acceptance Criteria**:

- **AC-110-1**: Given the AI engine generates a resume for a job application,
  When the generation completes successfully,
  Then a record is inserted into `resume_versions` with: application_id, job_title,
  company, resume_md_path, resume_pdf_path, match_score, llm_provider, llm_model,
  and created_at timestamp.

- **AC-110-2**: Given a resume version record exists,
  When the associated application is queried,
  Then the resume version can be retrieved by application_id.

- **AC-110-3**: Given a resume generation fails,
  When the error is caught,
  Then no resume version record is inserted.

**Negative Cases**:
- **AC-110-N1**: Given an application_id that does not exist in applications,
  When attempting to insert a resume version,
  Then the insert fails with a foreign key constraint error.

---

### FR-111: Resume Library API

**Description**: The system shall provide an API endpoint that returns a paginated
list of all resume versions with metadata and associated application status.

**Priority**: P0
**Source**: US-091, US-092
**Dependencies**: FR-110

**Acceptance Criteria**:

- **AC-111-1**: Given resume versions exist in the database,
  When `GET /api/resumes?page=1&per_page=50` is called,
  Then the response contains an array of resume version objects with fields:
  id, application_id, job_title, company, match_score, llm_provider, llm_model,
  created_at, application_status, resume_pdf_exists (boolean).

- **AC-111-2**: Given 75 resume versions exist,
  When `GET /api/resumes?page=2&per_page=50` is called,
  Then the response contains 25 items and total_count=75.

- **AC-111-3**: Given resume versions exist,
  When `GET /api/resumes?sort=created_at&order=desc` is called,
  Then results are sorted by creation date descending.

- **AC-111-4**: Given resume versions exist,
  When `GET /api/resumes?search=google` is called,
  Then only versions where company or job_title contains "google" (case-insensitive) are returned.

**Negative Cases**:
- **AC-111-N1**: Given no resume versions exist,
  When `GET /api/resumes` is called,
  Then the response contains an empty array and total_count=0.

- **AC-111-N2**: Given page=-1 is requested,
  When the API processes the request,
  Then page defaults to 1.

### FR-112: Resume Version Detail API

**Description**: The system shall provide an API endpoint that returns full details
of a single resume version, including the markdown content read from disk.

**Priority**: P0
**Source**: US-090, US-096
**Dependencies**: FR-110

**Acceptance Criteria**:

- **AC-112-1**: Given a resume version with id=5 exists and the markdown file is present,
  When `GET /api/resumes/5` is called,
  Then the response includes all metadata fields plus `resume_md_content` (string)
  with the full markdown text.

- **AC-112-2**: Given a resume version exists but the markdown file is missing from disk,
  When `GET /api/resumes/5` is called,
  Then the response includes all metadata but `resume_md_content` is null and
  `file_missing` is true.

**Negative Cases**:
- **AC-112-N1**: Given no resume version with id=999 exists,
  When `GET /api/resumes/999` is called,
  Then the response is 404 with error message.

- **AC-112-N2**: Given a resume version path contains `../`,
  When the API processes the request,
  Then path traversal is blocked and the file is not read.

---

### FR-113: Resume PDF Serve API

**Description**: The system shall provide an API endpoint that serves a resume PDF
file for inline viewing or download.

**Priority**: P1
**Source**: US-093, US-094
**Dependencies**: FR-110

**Acceptance Criteria**:

- **AC-113-1**: Given a resume version with id=5 exists and the PDF file is present,
  When `GET /api/resumes/5/pdf` is called,
  Then the response serves the PDF file with `Content-Type: application/pdf`.

- **AC-113-2**: Given the request includes `?download=true`,
  When the PDF is served,
  Then the response includes `Content-Disposition: attachment; filename="resume_5.pdf"`.

- **AC-113-3**: Given no download parameter,
  When the PDF is served,
  Then the response includes `Content-Disposition: inline` for in-app preview.

**Negative Cases**:
- **AC-113-N1**: Given the PDF file is missing from disk,
  When `GET /api/resumes/5/pdf` is called,
  Then the response is 404 with `{"error": "Resume PDF file not found"}`.

- **AC-113-N2**: Given the resume version path is outside the allowed directory,
  When the API processes the request,
  Then path traversal is blocked and 403 is returned.

### FR-114: Resume Effectiveness Metrics API

**Description**: The system shall provide an API endpoint that returns aggregate
effectiveness metrics for resume versions.

**Priority**: P0
**Source**: US-092
**Dependencies**: FR-110

**Acceptance Criteria**:

- **AC-114-1**: Given resume versions exist with various application statuses,
  When `GET /api/resumes/metrics` is called,
  Then the response includes:
  - `total_versions`: count of all resume versions
  - `tailored_interview_rate`: percentage of tailored resumes that led to interview/offer status
  - `fallback_interview_rate`: percentage of applications with fallback resume that led to interview/offer
  - `avg_score_interviewed`: average match_score for applications with interview/offer status
  - `avg_score_rejected`: average match_score for applications with rejected status
  - `by_provider`: array of {provider, count, interview_rate} per LLM provider used

- **AC-114-2**: Given interview statuses are "interview", "interviewing", "interviewed",
  and offer statuses are "offer", "accepted",
  When computing interview_rate,
  Then applications with any of these statuses count as positive outcomes.

**Negative Cases**:
- **AC-114-N1**: Given no resume versions exist,
  When `GET /api/resumes/metrics` is called,
  Then all rates are 0.0 and counts are 0.

### FR-115: Resume Library UI Screen

**Description**: The system shall display a dedicated Resume Library screen accessible
from the main navigation, listing all resume versions in a table/card format.

**Priority**: P0
**Source**: US-091
**Dependencies**: FR-111

**Acceptance Criteria**:

- **AC-115-1**: Given the user navigates to the Resume Library screen,
  When resume versions exist,
  Then a list/table displays: company, job title, match score, date generated,
  application status (as badge), and action buttons (view/download).

- **AC-115-2**: Given the user types in a search box,
  When the search term matches company or job title,
  Then the list filters to show only matching results.

- **AC-115-3**: Given the user clicks a column header,
  When the header is sortable (date, score, company),
  Then the list re-sorts by that column.

- **AC-115-4**: Given more than 50 resume versions exist,
  When the user scrolls to the bottom or clicks "Load More",
  Then the next page of results loads.

**Negative Cases**:
- **AC-115-N1**: Given no resume versions exist,
  When the user navigates to Resume Library,
  Then an empty state message is shown: "No resume versions yet. Resumes are generated
  automatically when you apply to jobs."

### FR-116: Resume Detail View UI

**Description**: The system shall display a resume detail view showing the rendered
resume content, metadata, and the associated application info.

**Priority**: P0
**Source**: US-090, US-096
**Dependencies**: FR-112, FR-113

**Acceptance Criteria**:

- **AC-116-1**: Given the user clicks a resume version in the library,
  When the detail view opens,
  Then it displays: rendered markdown content, job title, company, match score,
  generation date, LLM provider/model, and application status.

- **AC-116-2**: Given the detail view is open,
  When the user clicks "View PDF",
  Then the PDF renders in an embedded viewer (iframe/embed element).

- **AC-116-3**: Given the detail view is open,
  When the user clicks "Download PDF",
  Then the PDF downloads to the user's default download location.

- **AC-116-4**: Given the detail view is open,
  When the user clicks "View Application",
  Then the app navigates to the application detail view for the associated application.

**Negative Cases**:
- **AC-116-N1**: Given the resume files are missing from disk,
  When the detail view opens,
  Then a message displays: "Resume file not found on disk" with no preview.

**Description**: The system shall add a link from the existing application detail

**Priority**: P1

### FR-118: Resume Version Recording at Generation Time

**Description**: The system shall record resume version metadata at the time of
AI-generated resume creation, capturing the LLM provider and model used.

**Priority**: P0
**Source**: US-096
**Dependencies**: FR-110

**Acceptance Criteria**:

- **AC-118-1**: Given the AI engine generates a resume for a job application,
  When `generate_documents()` completes successfully,
  Then the function returns resume version metadata including llm_provider and llm_model
  in addition to the file paths.

- **AC-118-2**: Given the AI engine uses a fallback template (no LLM),
  When the fallback completes,
  Then no resume version record is created (fallback resumes are not versioned).

**Negative Cases**:
- **AC-118-N1**: Given the LLM call succeeds but PDF rendering fails,
  When the error is caught,
  Then no resume version record is created (partial generation is not recorded).

## 4. Non-Functional Requirements

### NFR-018-01: Query Performance

**Description**: Resume library API queries shall complete within 200ms at p95 for
databases with up to 10,000 resume versions.
**Metric**: p95 latency < 200ms for `GET /api/resumes` with 10K rows.
**Priority**: P0
**Validation Method**: Performance test with 10K inserted resume versions, measure query time.

### NFR-018-02: File I/O Safety

**Description**: All file read operations for resume content shall validate paths
against path traversal attacks and handle missing files gracefully.
**Metric**: Zero path traversal vulnerabilities. 100% graceful handling of missing files.
**Priority**: P0
**Validation Method**: Security audit + unit tests with malicious paths.

### NFR-018-03: Internationalization

**Description**: All user-visible strings in the Resume Library and Detail views
shall use `t()` (backend) or `t()`/`data-i18n` (frontend) for translation.
**Metric**: Zero hardcoded English strings in new code.
**Priority**: P0
**Validation Method**: Code review + grep for hardcoded strings.

### NFR-018-04: Accessibility

**Description**: All new UI components shall conform to WCAG 2.1 AA: semantic HTML,
ARIA attributes, keyboard navigability, visible focus indicators.
**Metric**: Zero WCAG 2.1 AA violations in new components.
**Priority**: P0
**Validation Method**: Manual accessibility review + automated checks.

### NFR-018-05: Test Coverage

**Description**: All new backend code shall have unit test coverage >= 80%.
All new API endpoints shall have integration tests.
**Metric**: >= 80% line coverage on new modules. >= 1 test per AC.
**Priority**: P0
**Validation Method**: pytest --cov report.

### NFR-018-06: Backward Compatibility

**Description**: Existing applications without resume version records shall continue
to function without errors. The resume library shall gracefully show "no data" states.
**Metric**: Zero regressions in existing test suite.
**Priority**: P0
**Validation Method**: Full test suite passes after implementation.

## 5. Interface Requirements

### 5.1 User Interfaces
- Resume Library screen: table/card list with search, sort, pagination
- Resume Detail view: metadata panel + rendered content + PDF embed + action buttons
- Application Detail view: additional "View Tailored Resume" button

### 5.2 Internal Interfaces
- `GET /api/resumes` — paginated list with search/sort
- `GET /api/resumes/<id>` — single version detail with markdown content
- `GET /api/resumes/<id>/pdf` — PDF file serve (inline or download)
- `GET /api/resumes/metrics` — effectiveness aggregate metrics

---

## 6. Data Requirements

### 6.1 Data Entities

**resume_versions** (new table):
- id: INTEGER PRIMARY KEY
- application_id: INTEGER FK → applications(id)
- job_title: TEXT NOT NULL
- company: TEXT NOT NULL
- resume_md_path: TEXT NOT NULL
- resume_pdf_path: TEXT NOT NULL
- match_score: INTEGER
- llm_provider: TEXT
- llm_model: TEXT
- created_at: DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

### 6.2 Data Retention
Resume version records persist as long as the application record exists.
Resume files on disk are user-managed.

### 6.3 Data Migration
No migration of existing data. New records created only for future resume generations.
Existing `applications.resume_path` continues to work independently.

## 7. Out of Scope

- **Side-by-side diff comparison**: Requires markdown parsing + diff algorithm. Deferred to v2.2.0.
- **Resume editing in-app**: Out of product scope. Users regenerate via bot.
- **Tagging/starring/favoriting**: Deferred to v2.2.0. Requires additional DB column.
- **Cover letter versioning**: Separate feature, same pattern. Future backlog.
- **Resume search by content**: Full-text search of resume body. Deferred.
- **Batch export/ZIP download**: Deferred to v2.2.0.
